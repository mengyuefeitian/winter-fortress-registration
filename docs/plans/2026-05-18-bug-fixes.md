# Bug Fix Implementation Plan

## Overview

Fix 10 reported bugs across canyon/arsenal registration, statistics pages, config deletion, and zone management.

## Bug Root Causes & Fixes

### Bug 1: 峡谷会战报名列表统计显示 0

**Root Cause**: `miniprogram/pages/user/canyon-registration/canyon-registration.js` `loadConfigs()` (line ~190) queries `arsenalRegistrations` collection with `activityType: 'canyon'` filter. But the `arsenalRegistrations` collection stores `arsenal` type registrations — canyon registrations go to `canyonRegistrations` collection. The cloud function `manageArsenal` uses `getCollectionNames(activityType)` to map `canyon` → `canyonRegistrations`, but the client-side query bypasses the cloud function and queries `arsenalRegistrations` directly.

**Fix**: In `canyon-registration.js` `loadConfigs()`, change the direct DB query from `arsenalRegistrations` to `canyonRegistrations` collection. The query currently uses:
```javascript
const res = await wxdb.collection('arsenalRegistrations').where({
  configId: wxdb.command.in(configIds),
  activityType: 'canyon',
  status: 'active'
})
```
Should query `canyonRegistrations` without the `activityType` filter (since canyonRegistrations only has canyon data).

**Files**: `miniprogram/pages/user/canyon-registration/canyon-registration.js`

### Bug 2: 昵称重复报错信息为英文 "ResourceUnava"

**Root Cause**: The cloud database transaction rollback in `manageArsenal/index.js` `createRegistration()` throws `Error('该昵称已被使用')` which is Chinese. But the error from the cloud database layer (when the transaction rolls back) may return a raw error like "ResourceUnavailable" that bubbles up. The `db.js` wrapper functions catch `res.result.error` but may also catch the raw cloud error.

**Fix**: In `db.js` `createArsenalRegistration` and `createCanyonRegistration`, catch the error and check if the message contains "Resource" or "Unava" — replace with a user-friendly Chinese message. Also in the cloud function `manageArsenal/index.js`, wrap the transaction error handling to ensure Chinese messages pass through cleanly.

**Files**: 
- `miniprogram/utils/db.js` (createArsenalRegistration, createCanyonRegistration)
- `miniprogram/cloudfunctions/manageArsenal/index.js` (createRegistration error handling)

### Bug 3: 分区管理中联盟数量显示为 0

**Root Cause**: `miniprogram/pages/admin/zone-manage/zone-manage.js` (line ~80) queries alliances with `status: 'active'` filter. Historical alliances may not have a `status` field, so they're excluded. Also the `.get()` without pagination could miss alliances if there are more than 20 per zone (unlikely with max 12, but the `status` filter is the main issue).

**Fix**: Remove the alliance count logic entirely from zone-manage (user requested). Remove the "联盟数量" row from wxml and the counting code from JS.

**Files**:
- `miniprogram/pages/admin/zone-manage/zone-manage.wxml` (lines 36-39)
- `miniprogram/pages/admin/zone-manage/zone-manage.js` (alliance counting logic)

### Bug 4: 兵工厂报名列表显示 "arsenal" 标签 + 参战/替补未分开

**Root Cause (label)**: `miniprogram/pages/user/arsenal-registration/arsenal-registration.wxml` line 79 uses `{{ACTIVITY_TYPE_LABELS[item.activityType] || item.activityType}}`. The `ACTIVITY_TYPE_LABELS` must be defined in the page's JS `data` object for wxml to access it. If it's not in `data`, the wxml can't resolve it and falls back to `item.activityType` ("arsenal").

**Root Cause (stats)**: `arsenal-registration.wxml` line 86 shows `{{item.totalCount}}/{{item.totalCapacity}}人` — combined count. The JS already computes `combatCount` and `substituteCount` per config (line 234-256), but the wxml doesn't display them separately.

**Fix**: 
1. Add `ACTIVITY_TYPE_LABELS` to the page's `data` object in `arsenal-registration.js`
2. Update wxml to show `参战 {{item.combatCount}}/{{30}} 替补 {{item.substituteCount}}/{{10}}` (same pattern as canyon-registration.wxml)

**Files**:
- `miniprogram/pages/user/arsenal-registration/arsenal-registration.js`
- `miniprogram/pages/user/arsenal-registration/arsenal-registration.wxml`

### Bug 5: 区管/盟管删除峡谷活动失败 — "缺少必要参数"

**Root Cause**: `db.deleteArsenalConfig` (db.js line 1439) and `db.deleteCanyonConfig` (db.js line 1493) only pass `{ configId }` to the cloud function. The cloud function `manageArsenal/index.js` `deleteConfig` (line 270-274) requires `{ configId, activityType }`:
```javascript
if (!configId || !activityType) {
  throw new Error('缺少必要参数')
}
```

**Fix**: Add `activityType` to both delete function calls:
- `deleteArsenalConfig`: pass `{ configId, activityType: 'arsenal' }`
- `deleteCanyonConfig`: pass `{ configId, activityType: 'canyon' }`

**Files**: `miniprogram/utils/db.js` (lines 1439, 1493)

### Bug 8: 403 Image Error

**Root Cause**: Cloud storage URL tokens expire. The avatar URL includes a `sign` parameter and `t=` timestamp that have a limited validity period.

**Fix**: Not a code bug. This is expected behavior when cloud storage URLs expire. The app should use a cloud function to serve avatar URLs dynamically or use permanent URLs. Not critical for this fix session.

### Bug 9: Timeout Error

**Root Cause**: General network timeout, not a code bug. Could be caused by slow cloud function execution, poor network, or cloud environment issues.

**Fix**: Not a code bug. Monitor and investigate if it becomes persistent.
